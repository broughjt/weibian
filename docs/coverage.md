Strong signal, with three caveats worth flagging.

**Why it's a strong signal.** The property is *behavioral equivalence to a much simpler reference*, executed across 80k random sequences of up to 40 mutations each, on a small id pool that deliberately produces collisions. Any incremental-only bug — stale dirty propagation, missed cache eviction, wrong delete semantics, backmatter that doesn't refresh when its target flips — has to make the two implementations disagree, and proptest will shrink and report it. The fact that nothing diverged after dropping `metadata_dirty` is a direct empirical answer to "did losing that axis cause silent staleness?": no, the dirty-only tracking is sufficient because every operation that changes title/metadata also dirties the id.

**The three caveats.**

1. **The reference compiler isn't ground truth.** It's also code I read carefully today, but if both implementations share a blind spot (e.g., both miscount something in cycle-with-duplicate interactions in the same way), the proptest is happy. For the parts of stage 1 I lifted directly from `process_stateless.rs` — dangling-target detection, the cycle-diagnostic emission shape — the proptest can only confirm I copied them faithfully, not that they're right. The four hand-written `duplicate_identifier_*` tests are doing real work here as the absolute-correctness counterweight.

2. **No coverage measurement.** I don't know which lines of `process_stage1` get exercised most. The strategy is well-tuned for collisions (small id pool), but specific paths are still rare under random generation: e.g., a node that transcludes a duplicate *and* is itself in a cycle, or removing the last file of a duplicate cluster while another file is mid-update. Those paths exist in my code; the proptest probably hits them, but I can't tell you how often. A `cargo llvm-cov` pass over a 1k-case run would close this — quick to add, high information value.

3. **What the property doesn't observe.** The test compares filesystem state and diagnostics — it doesn't compare in-memory caches. So a leak in `rendered_bodies` (id stays cached after the node was deleted) is invisible to it. I added cache eviction in three delete paths in stage 1, but the proptest gives me zero confidence those are right; it'd happily pass with all the eviction code commented out, as long as the filesystem ends up correct. If you care about memory hygiene under long-running watch sessions, it's worth a separate assertion (e.g., a debug-only invariant check that `rendered_bodies.keys()` is a subset of singleton ids in `self.nodes`).

Net: I'd ship this. But I'd add the cache-hygiene invariant as a debug assertion in `_process` (cheap, catches a class of bugs the property test structurally cannot), and run llvm-cov once just to spot any obvious untested branches.
